# แพตเทิร์นการออกแบบ Scenario (Blind Protocol)

เอกสารนี้สรุปโครงสร้างที่ใช้คิด `situation` + เงื่อนไขชนะ/แพ้ ให้อ่านและต่อยอด scenario ใหม่ได้สม่ำเสมอ

---

## สองชั้นของ “ภารกิจ”

| ชั้น | ความหมาย | ตัวอย่าง (ร้านทองถูกปล้น) |
|------|-----------|---------------------------|
| **Rules** | กติกาของสถานการณ์ — **ทำอย่างไรถือว่าแพ้ทันที** หรือขีดจำกัดที่ต้องไม่ให้เกิด | ต้องปกป้องทรัพย์สิน: **โจรห้ามได้ของแล้วหนีไปได้** |
| **Goals** | เป้าหมายเชิงบวก — **ทำอย่างไรถือว่าชนะ** (อาจมีมากกว่าหนึ่งทาง) | เช่น **จบวิกฤตโดยทรัพย์ไม่ถูกปล้นสำเร็จ** — อาจชนะ goal ได้แม้ตัวเราจะไม่รอด (ถ้าเรื่องนั้นออกแบบไว้) |

ใน `situation` ควรเขียนให้เห็นทั้งสองชั้น (หรือ rules ชัด + goal ชัด) เพื่อให้ผู้เล่นและ GM (AI) ใช้ทิศทางเดียวกัน

---

## ผลลัพธ์ (จับ pattern เป็นเงื่อนไข)

แยกเป็น **แพ้** vs **ชนะ** ตาม rules ก่อน แล้วค่อยดู goal

### แพ้ (fail)

- เกิดเหตุที่ **rules ห้าม** — เช่น โจรได้ของแล้วหนีไปได้ = **fail**

### ชนะ (success)

อย่างใดอย่างหนึ่ง (แล้วแต่เราออกแบบให้เทียบเท่ากัน):

- **ชนะทาง rules:** ภัยถอย / ปล้นไม่สำเร็จ — เช่น โจรต้องล่าถอยไปเอง ไม่ได้ของหรือไม่หนีพร้อมของตามที่นิยาม “สำเร็จ” ของโจร
- **ชนะทาง goal:** เป้าหมายหลักสำเร็จ — เช่น **ภัยคลายและการปล้นล้มเหลว** (ไล่โจรได้ เจ้าหน้าที่เข้ามาคุมจนภัยสิ้น ฯลฯ — ตามที่ `situation` นิยาม) ก่อนจบแบบแพ้ rules — **ไม่นับแค่ “แจ้งเหตุ / โทรออกไป” ถ้าเล่ายังไม่ยืนยันว่าวิกฤตจบและทรัพย์ปลอดภัยจริง**

หมายเหตุ implementation: ผลจบเกมอ่านจาก `worldState` — **`rule_*` = `yes` แพ้ทันที**; **ชนะ** เมื่อ **ไม่แพ้ rule** และ (**มี `goal_*` ใด ๆ เป็น `yes`** หรือ `mission_success=yes` เป็น fallback); **`[MISSION_IMPOSSIBLE]`** จาก GM → แพ้; phase `end` เมื่อหมดรอบเทิร์นโดยยังไม่ชนะ → แพ้

### การตาย vs การชนะ — ไม่ผูกกันเสมอไป

- **ตัวละครหลัก “เล่นไม่ได้”** (`system_protagonist_alive` = `no`) ไม่ได้แปลว่า “แพ้” อัตโนมัติ ถ้า **เป้าหมายหลักสำเร็จแล้ว** — เช่น วิกฤตจบและทรัพย์ไม่ถูกปล้นสำเร็จในจังหวะเดียวกับที่เราล้ม **เรื่องร้านทองอาจยังถือว่าชนะภารกิจ** แม้ตัวเราจะไม่รอด — หลัง `no` เซิร์ฟเวอร์จะ **เล่าเรื่องต่ออัตโนมัติ** (ไม่รับ input จากผู้เล่น) จนกว่าจะจบรอบ
- ออกแบบ scenario ให้ชัดว่า **ชนะวัดที่ goal หรือที่การรอดชีวิต** หรือทั้งสองมีความหมายคนละอย่าง
- เครื่องยนต์เกม: **`isRuleFailed`** (มี `rule_*` = `yes`) → แพ้; ไม่เช่นนั้น **`isMissionWon`** (`!isRuleFailed` และมี `goal_*` = `yes` หรือ `mission_success=yes`) — ตรวจ **ก่อน** ถือว่าแพ้เพราะความตาย / หมดรอบเทิร์น

## เชื่อมกับไฟล์ข้อมูล

| ส่วน | บทบาท |
|------|--------|
| `situation` | **ฉากเล่า** ที่ **ฝังสภาพเริ่มต้นที่สำคัญไว้ในประโยค** (ให้สอดคล้อง `worldState` โดยตรง) — เขียนเป็น**จังหวะเรื่อง** (ภัย คอนทราสต์ภายใน–ภายนอก แล้วรายละเอียดในฉาก) ไม่ใช่ไล่อ่าน key ทีละบรรทัด; ยังคงสั้น ตรง ๆ ได้ |
| `worldState` | **snapshot ตอนเริ่มเกม** (initial) — รวม **กติกา/เป้าหมาย** (`rule_*` / `goal_*`) และ **ระบบ** (`system_*`) กับ **ทุกสิ่งที่ตัวเอกรับรู้ได้ในฉาก ณ จุดเริ่มต้น** (ดูด้านล่าง); GM ใช้ต่อยอดระหว่างเล่น |
| `themes` | แท็กหา scenario / โทนเรื่อง — ไม่ใช่กติกา |

ตอนนี้ `data/scenarios.json` ใน repo ใช้ **ฉากร้านทอง** เป็นตัวอย่างหลักที่สอดคล้องตารางและ checklist ด้านล่าง

### worldState กับสิ่งที่ตัวเอกรับรู้ได้

- **ชุด initial ใน `scenarios.json`** — ควร **ครบทุกอย่างที่ตัวเอกรับรู้ได้** ณ เวลาเริ่มเกม (snapshot เปิดฉาก) เพื่อให้ GM ไม่ต้องเดา; รายการที่ไม่เกี่ยวกับ opening ไม่จำเป็นต้องใส่
- **ฉาก โจร ฯลฯ** — ควรเป็นสิ่งที่ตัวเอกรับรู้ได้ผ่าน **การมอง ฟัง สัมผัส สิ่งที่ปรากฏ** — ไม่ใช่ความคิดในใจของคนอื่นโดยตรง
- ความ “หัวร้อน” จึงควรเป็น **รูปแบบที่สังเกตได้** เช่น น้ำเสียง ท่าที การขู่ — ใช้ key อย่าง `robbers_perceived_demeanor` ค่า `agitated`, `threatening`, `nervous` ฯลฯ ไม่ใช่ meter อารมณ์ที่ตัวเอกรู้ไม่ได้
- **ข้อยกเว้น**: `system_*`, `rule_*`, `goal_*` เป็น **ข้อตกลงของเกม** สำหรับเครื่องยนต์/AI — ไม่จำเป็นต้องเป็นสิ่งที่ตัวเอก “รู้” ในโลกเรื่องแบบเดียวกับเสียงโจร

### ภาษาใน `situation` (สิ่งที่ควรระวัง)

- **ปิดท้ายด้วยภารกิจ** — พูดเป็น**ผลที่ตรวจได้** (เช่น “แพ้เมื่อโจรหนีพร้อมของ” / “ชนะเมื่อจบวิกฤตโดยการปล้นไม่สำเร็จ”) ให้สอดคล้อง `rule_*` / `goal_*` มากกว่าคำกว้างที่ไม่บอกเกณฑ์
- **ชนะ vs แค่เริ่มขั้นตอน** — ถ้าฉากมีปุ่มแจ้งเหตุ/โทรศัพท์ ควรเขียนให้ GM รู้ว่า **การแจ้งหรือโทรอย่างเดียวไม่จำเป็นต้องเท่ากับชนะ** ถ้านิยามชนะคือ “ภัยสิ้น / ปล้นล้มเหลวจริง” — ไม่เช่นนั้น Layer C อาจตั้ง `goal_*` เร็วเกินไปจากแค่ “กดแจ้งแล้ว”
- **ไม่ใส่ประโยคเชื่อมที่ไม่เพิ่มภาพ** — ถ้าตัดแล้วความหมายไม่หาย ให้ตัด
- **ไม่ใส่กรอบรวมแล้วค่อยไล่รายการ** — พอสร้างบรรยากาศ/คอนทราสต์แล้ว ต่อด้วยรายละเอียดที่สัมผัสได้ได้เลย ไม่ต้องมีวลีคลุมเช่น “ทุกอย่างยังอยู่ในการควบคุม” ถ้าจะไล่รายการอยู่แล้ว

### ตัวอย่างอ้างอิง: ร้านทอง (initial)

ใช้เป็นแม่แบบว่า `situation` กับ `worldState` **สะท้อนกัน**อย่างไร (ไม่ต้องคัดลอกข้อความทั้งก้อนไปทุก scenario)

| จังหวะในเรื่อง | สอดคล้องกับ state (หมวด) |
|----------------|---------------------------|
| เปิดร้านตามปกติ แล้วโจรเข้ามาในห้องขาย | `robbers_presence`, `shop_front_entry` |
| ไม่มีลูกค้าคนอื่น | `shop_civilian_presence` |
| ภัยชัด เสียงก้าว ปืนในมือ ท่าทีกดดัน | `robbers_perceived_demeanor`, `robbers_visible_threat` |
| ข้างนอกได้ยินรถผ่านถนน | `shop_street_audible` |
| ในร้านไฟสว่าง ทองหลังกระจก ตู้โชว์ กระจกหน้าร้าน เซฟ ทางหลัง ห้องหลัง โทรศัพท์ ปุ่มแจ้งเหตุ ระบบยังเงียบ โจรยังไม่เอาทองหนี | `shop_*` ที่เกี่ยวกับแสง กระจก ตู้ ประตู ปุ่ม แจ้งเหตุ โทรศัพท์ + `rule_robbers_escaped_with_loot` |
| ปิดท้ายภารกิจ (ชนะ = วิกฤตจบ ปล้นไม่สำเร็จจริง — ไม่ใช่แค่แจ้ง/โทร) | `rule_*` / `goal_crisis_resolved_without_loot_loss` (พูดให้ตรงกับคำนิยามใน `situation`) |

ชุด key ตัวอย่างใน `worldState` (initial) สำหรับร้านทอง — ครอบคลุมสิ่งที่ตัวเอกรับรู้ได้: `system_protagonist_alive`; `rule_robbers_escaped_with_loot`; `goal_crisis_resolved_without_loot_loss`; `robbers_perceived_demeanor`, `robbers_presence`, `robbers_visible_threat`; `shop_back_office_door`, `shop_civilian_presence`, `shop_display_case_front`, `shop_display_glass`, `shop_front_entry`, `shop_front_window_glass`, `shop_landline_phone`, `shop_lights`, `shop_rear_emergency_exit`, `shop_silent_alarm`, `shop_silent_alarm_button`, `shop_street_audible`, `shop_vault_room_door`

### โครงสร้าง `worldState`: แยกด้วย prefix (ไม่ต้องแยก object ใน JSON)

โค้ดเกมใช้ `worldState` เป็น object แบน (flat) ชุดเดียว — **ไม่จำเป็นต้องแยกเป็น `rulesState` / `goalsState` คนละ object** ถ้ายังไม่ต้องการเปลี่ยน schema ทั้งระบบ

แนะนำให้ **แยกด้วยชื่อ key** ให้อ่านรู้บทบาททันที:

| Prefix / กลุ่ม | ใช้เก็บอะไร | ตัวอย่าง (ร้านทอง) |
|----------------|-------------|---------------------|
| `system_*` | **ระดับระบบเกม** — ไม่ใช่กติกาเรื่อง (rule) | `system_protagonist_alive`: `yes` = ผู้เล่นยังบังคับตัวละครหลักได้; `no` = เล่นไม่ได้แล้ว — log ต่อด้วยระบบอัตโนมัติ (ไม่มี input) |
| `rule_*` | สถานะที่สะท้อน **rules** ของเรื่อง — เงื่อนไขแพ้หรือ “เส้นตาย” | `rule_robbers_escaped_with_loot`: `no` → ถ้าเป็น `yes` แปลว่าแพ้ตามกติกา |
| `goal_*` | สถานะที่สะท้อน **goals** — เป้าหมายเชิงบวกที่ติดตามได้ | `goal_crisis_resolved_without_loot_loss`: `no` / `yes` (ฉากร้านทอง — ชนะเมื่อเล่าชัดว่าภัยคลายและปล้นไม่สำเร็จจริง **ไม่ใช่แค่แจ้งเหตุหรือโทรแล้วจบ**) |
| `shop_*` (หรือ prefix สถานที่อื่นตามเรื่อง) | สภาพกายภาพของฉาก — ประตู กระจก แสง ปุ่ม เสียงข้างนอก คนอื่นในร้าน โทรศัพท์ ฯลฯ | เช่น `shop_vault_room_door`, `shop_display_glass`, `shop_front_window_glass`, `shop_civilian_presence`, `shop_street_audible`, `shop_landline_phone`, `shop_silent_alarm_button` |
| `robbers_*` (หรือ `antagonist_*` ตามเรื่อง) | **ตัวร้าย / คู่ปฏิบัติการ** — สิ่งที่ตัวเอก**รับรู้ได้** (อยู่ที่ไหน ท่าที ภัยที่มองเห็น ฯลฯ) — คนละอย่างกับ `rule_*` ที่บอกว่า “แพ้แล้ว” หรือยัง | `robbers_presence`, `robbers_perceived_demeanor`, `robbers_visible_threat` (เช่น `firearm_visible`) — สังเกตจากภายนอก ไม่ใช่ meter ในใจโจร |

โค้ดจะเติม `system_protagonist_alive: "yes"` ตอนโหลด scenario ถ้าไฟล์ไม่ใส่

ถ้าอนาคตอยาก **แยก object ในไฟล์ scenario** (เช่น `rulesState` / `goalsState` แล้ว merge เข้า `worldState` ตอนโหลด) ทำได้ — แต่สำหรับการออกแบบและ AI ตอนนี้ **prefix ใน object เดียว** มักพอและซิงก์กับ `Room.worldState` ได้ตรง ๆ

## checklist ตอนออกแบบ scenario ใหม่

1. เขียน **rules** ว่า “อะไรแล้ว = แพ้”
2. เขียน **goals** ว่า “อะไรแล้ว = ชนะ” (มีกี่ทางบ้าง)
3. ตรวจว่าไม่ขัดกันระหว่างชนะทาง A กับแพ้ทาง rules
4. ใส่ใน `worldState` ทั้ง **rule_*/goal_*** (ถ้าจับเป็นค่าได้) และ **สภาพแวดล้อม** — ใช้ prefix ให้สอดคล้อง rules/goals
5. ตรวจว่า **`situation` กับ `worldState` เริ่มต้นสะท้อนกัน** — ถ้า state บอก “ล็อก” สถานการณ์ต้องไม่บอกว่าเปิดแล้ว; ถ้า `goal_*` ยัง `no` ต้องไม่เขียนว่าชนะตามนิยามแล้ว (เช่น วิกฤตจบแล้ว)
6. เขียน `situation` แบบ **ฝังรายละเอียดที่จำเป็นในภาษาเล่า** (เช่น “ทองยังอยู่หลังกระจกที่ล็อก”) แทนการไล่ key ทีละบรรทัด
7. สำหรับคีย์ฉาก/โจร — ใส่เฉพาะสิ่งที่ตัวเอก**รับรู้ได้**; ไม่ใส่ความรู้ภายในของโจรโดยตรง
8. ข้อความปิดท้ายใน `situation` — ใช้ภาษาพูด**ตรงกับกติกา** (แพ้เมื่ออะไร / ชนะเมื่ออะไร — เช่น จบวิกฤตโดยปล้นไม่สำเร็จ) แทนคำกว้างที่ซ้ำความหมาย; ถ้ามีช่องทาง “แจ้งเหตุ” ให้ชัดว่า **ชนะต้องเป็นผลจริง** ไม่ใช่แค่เริ่มขั้นตอน

---

*อัปเดตล่าสุด: สะท้อน `data/scenarios.json` (ฉากร้านทอง) — `rule_robbers_escaped_with_loot=yes` = แพ้; `goal_crisis_resolved_without_loot_loss=yes` (หรือ `mission_success=yes` เมื่อชนะชัดแต่ไม่ map กับชื่อ goal) = ชนะเมื่อยังไม่แพ้ rule; success ทางเล่า = ภัยคลายและการปล้นไม่สำเร็จ (ไล่โจร เจ้าหน้าที่คุมจนภัยสิ้น ฯลฯ ตาม `situation`) **ไม่ใช่แค่แจ้งเหตุหรือโทรแล้วจบ** — ให้ GM อัปเดต state ให้ตรง*
